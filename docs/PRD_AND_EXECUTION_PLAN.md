# AIGC 视频项目协同工作台 PRD 与执行方案

## 1. 项目定位

本系统是面向内部团队的 AIGC 视频项目协同工作台，用于承接未来交付给甲方的 AIGC 视频生成业务。第一阶段完整落地商业业务前四步，后续逐步扩展到创意表达实现与交付归档全流程。

第一阶段必须达到生产可用标准：真实创建项目、真实上传资料、真实调用 AI、真实保存状态、真实生成产物、真实飞书交付，不接受静态 mock 前端替代核心链路。

## 2. 一期目标

一期完整实现以下四个环节：

1. 品牌方需求洽谈
2. 技术可行性评估
3. 创意方向提案
4. 方向初选、报价与签约

5-12 步先作为全流程阶段导航和状态预留，不实现具体业务能力：

5. 完整脚本与深化
6. 人物与场景视觉设定
7. 文字分镜拆解
8. 分镜图片生成与确定
9. 视频生成与素材筛选
10. A copy 生成与修改
11. B copy 与定稿确认
12. 结算交付与归档

### 3.1 商务团队仪表盘

商务团队负责客户沟通、项目创建、需求收集、资料上传、报价合同、飞书交付和客户确认跟进。

核心模块：

- 待推进项目
- 待补充需求
- 待确认报价
- 待确认合同
- 待飞书发送
- 已发送记录
- 客户联系人与群聊记录

### 3.2 创意团队仪表盘

创意团队负责样片分析结果查看、创意方向筛选、方向深化、故事大纲或梗概、氛围图生成与修改。

核心模块：

- 待创意评估项目
- Top 5 创意方向
- 待深化方向
- 待生成氛围图
- 待人工修改内容
- 已确认创意资产

### 3.3 管理团队仪表盘

管理团队包括管理员角色，负责项目全局进度、权限、评分规则、模型配置、成本、异常任务和归档审计。

核心模块：

- 全局项目看板
- 阻塞项目
- 阶段进度统计
- 评分规则配置
- 模型与 provider 配置
- AI 调用成本
- 异常任务
- 操作审计

## 4. 核心工作台结构

系统采用双栏工作台。

左侧为项目列表，类似 AI chat 的项目会话列表。每个项目必须展示：

- 品牌名
- 项目名
- 当前阶段
- 负责人
- 截止时间
- 状态

右侧为当前项目工作台，根据项目状态机自动展示当前阶段的业务界面。例如：

- 当前阶段是需求洽谈时，展示标准需求模板、资料上传和样片标签。
- 当前阶段是技术可行性评估时，展示评分结果、标签命中和 Top 5 创意方向。
- 当前阶段是创意方向提案时，展示创意卡片、方向深化和氛围图。
- 当前阶段是报价签约时，展示提案、报价、合同富文本和飞书交付。

顶部或右侧需要展示 12 步阶段导航，每一步都有明确状态。

## 5. 一期主流程

项目创建 -> 需求资料上传 -> Agent 结构化需求 -> 样片/图片/视频理解 -> 自动打标签 -> 评分规则计算 -> Top 5 创意卡片 -> 人工单选或多选 -> 每个方向扩展 4-5 个故事大纲或梗概 -> 根据故事大纲生成氛围图 -> 人工编辑 -> 生成提案、报价、合同 -> 人工确认 -> 创建飞书文档 -> 发送给甲方联系人或群 -> 回写发送记录 -> 项目归档。

### 5.1 异常与回退流程

- 客户资料不足：项目进入 `needs_revision`，商务团队补充资料后重新提交。
- 技术不可行：项目进入 `blocked`，必须记录不可行原因，不能直接进入报价签约。
- 样片分析失败：资产分析任务进入 `failed`，允许重试或重新上传文件。
- 创意方向被驳回：方向进入 `needs_revision`，创意团队修改后重新提交。
- 报价被驳回：报价保存新版本，项目回到报价编辑状态。
- 合同被驳回：合同保存新快照，项目回到合同编辑状态。
- 飞书发送失败：交付记录进入 `failed`，允许重试或人工复制交付。
- 项目取消：项目进入 `archived` 或终止状态，保留操作日志和已有产物。

### 5.2 成功指标

一期上线后建议跟踪：

- 需求结构化平均耗时
- 技术评估平均耗时
- Top 5 创意方向生成成功率
- 创意方向一次通过率
- 报价合同生成平均耗时
- 飞书交付成功率
- AI 任务失败率与重试成功率
- 每个项目人工修改次数
- 每个项目 AI 调用成本

## 6. 功能需求

### 6.1 项目管理

- 支持创建、编辑、查看项目。
- 支持设置品牌名、项目名、当前阶段、负责人、截止时间、状态。
- 支持按角色展示项目待办。
- 支持项目阶段流转。
- 支持项目状态刷新后恢复，不允许刷新页面导致进度丢失。

### 6.2 需求资料中心

一期必须支持以下输入：

- 人工粘贴文本
- PDF 文件
- Word 文件
- 视频文件
- 图片文件
- 飞书文档链接

所有输入需要统一进入资产表，关联项目、上传人、文件类型、解析状态和 OSS 地址。

### 6.3 需求结构化

使用豆包文本模型将客户需求整理成统一需求模板。

模型：

```env
ARK_TEXT_STRUCTURING_MODEL=doubao-seed-2-1-pro-260628
```

标准需求模板至少包含：

- 品牌信息
- 产品或服务信息
- 目标受众
- 视频目标
- 期望风格
- 参考样片
- 核心卖点
- 禁忌点
- 交付规格
- 时间节点
- 预算或报价信息
- 待确认问题

### 6.4 样片与素材分析

系统需要自动分析视频、图片和样片，识别风格、画面、节奏、质感、技术难点和可复用标签。

模型：

```env
ARK_IMAGE_VIDEO_UNDERSTANDING_MODEL=doubao-seed-2-0-lite-260215
```

业务重点视频风格：

- 三维风格
- 写实风格

### 6.5 标签与评分规则后台

一期评分规则后台支持：

- 配置 tag
- 配置 tag 权重
- 配置评分描述
- 配置正向样例
- 配置负向样例

评分规则用于将客户需求和样片分析结果落入素材库标签体系，并生成可解释分数。

### 6.6 技术可行性评估

系统根据标签、评分规则和素材库语义检索，输出 Top 5 创意方向卡片。

模型：

```env
ARK_MATERIAL_EMBEDDING_MODEL=doubao-embedding-vision-251215
```

每张卡片必须包含：

- 标题
- 核心创意
- 适配理由
- 风险提示
- 参考标签
- 综合评分
- 成本预估
- 周期预估
- 技术难度
- 氛围图占位或结果
- 详情页
- 单选/多选状态
- 人工可编辑内容

### 6.7 创意方向提案

用户可对 Top 5 创意方向进行单选或多选。系统根据选择方向继续深化，每个方向生成 4-5 个故事大纲或梗概。

每个故事大纲或梗概需要包含：

- 标题
- 一句话概念
- 起承转合
- 核心视觉亮点
- 适合的画面风格
- 预估制作难度
- 风险提示

### 6.8 氛围图生成

根据故事大纲或梗概生成匹配方向的氛围图。

模型：

```env
ATMOSPHERE_IMAGE_PROVIDER=openai
OPENAI_IMAGE_MODEL=gpt-image-2-all
```

必须保存：

- 生成任务状态
- prompt
- 模型名
- 生成结果
- OSS 地址
- 失败原因
- 重试次数
- 人工确认状态

### 6.9 提案、报价与合同

系统需要支持：

- 提案富文本编辑
- 报价内容编辑
- 合同模板字段填充
- 甲方合同上传
- 合同富文本编辑
- 历史快照
- 导出 PDF
- 导出 Word
- 人工确认

历史快照不要求严格 diff 审计，一期保存快照即可。

### 6.10 飞书交付

系统内编辑提案或合同后，人工确认并触发飞书交付。

交付闭环：

1. 创建飞书文档
2. 发送给甲方个人或群
3. 回写飞书文档链接
4. 回写发送对象
5. 回写发送时间
6. 回写发送结果
7. 进入项目归档记录

## 6.11 数据留存与归档

- 客户资料、样片、AI 生成资产、提案、报价、合同和飞书交付记录默认随项目长期保留。
- 删除项目不应立即物理删除资产，优先进入归档或软删除状态。
- 管理团队可以查看归档项目和关键操作日志。
- 合同、报价和飞书交付记录属于敏感数据，必须受权限控制。
- 下载、预览、导出敏感文件需要记录操作日志。

## 7. 状态机要求

所有项目进度必须由后端状态机驱动，不能只依赖前端临时状态。

阶段状态：

```text
not_started
in_progress
waiting_review
needs_revision
approved
blocked
completed
archived
```

AI 任务状态：

```text
queued
processing
succeeded
failed
retrying
cancelled
```

报价与合同状态：

```text
draft
waiting_review
needs_revision
confirmed
sent
signed
terminated
```

每个阶段都需要保存：

- 当前状态
- 负责人
- 开始时间
- 完成时间
- 输入材料
- 产出物
- 人工确认记录
- 错误信息
- 重试次数
- 历史快照

刷新页面、重新打开系统、重新登录或换设备后，系统必须从数据库恢复项目当前状态、阶段产物和任务进度。

## 8. 自然语言提示与错误处理

所有前端提示和报错必须使用自然语言，不直接暴露技术栈错误、堆栈、HTTP 原始错误或模型原始异常。

示例：

- 样片分析失败，可能是视频格式或文件过大导致。你可以重新上传 MP4，或点击重试。
- 飞书文档已创建，但发送给联系人失败。请检查联系人是否存在，或改为发送到群。
- 合同已保存为新版本，你可以继续编辑或恢复上一版。
- 氛围图生成失败，当前模型没有返回可用图片。你可以调整方向描述后重新生成。

服务端日志需要保留可诊断信息，但用户侧不得暴露堆栈、密钥、原始 SDK 异常或数据库错误。

## 9. 技术架构

前端：

- Next.js
- React
- TypeScript
- shadcn/ui

后端：

- Node.js / Next.js API 服务
- 部署到阿里云

数据库：

- Postgres 兼容托管数据库
- 优先选择成本可控的 Neon 或 Supabase

文件存储：

- 阿里云 OSS

- 代码层保留 asset provider 抽象，便于未来替换第三方云存储

异步任务：

- AI 分析
- 视频/图片理解
- 氛围图生成
- 文档导出
- 飞书发送

AI provider：

- 火山方舟豆包
- OpenAI 图像生成

飞书：

- Feishu OpenAPI

富文本：

- Tiptap 或 Lexical

导出：

- PDF
- Word

## 14. 一期验收标准

一期完成时必须满足：

- 可以真实创建项目
- 可以真实上传资料到 OSS
- 可以真实保存和恢复项目状态
- 可以真实调用豆包完成需求结构化
- 可以真实调用豆包完成图片/视频理解和标签落入
- 可以真实配置评分规则
- 可以真实生成 Top 5 创意方向
- 可以真实深化每个方向的故事大纲或梗概
- 可以真实调用 `gpt-image-2-all` 生成氛围图
- 可以人工编辑提案、报价和合同
- 可以保存合同历史快照
- 可以导出 PDF/Word
- 可以真实创建飞书文档
- 可以真实发送给甲方个人或群
- 可以回写飞书交付记录和归档链接
- 页面刷新或重新打开后当前进度不丢失
- 所有失败、加载、空状态都有自然语言反馈

### 14.1 端到端验收路径

至少完成并验证一条完整路径：

1. 商务团队创建新项目。
2. 商务团队上传文本需求、PDF/Word、图片或视频资料。
3. 系统将资料保存到 OSS，并在数据库记录资产状态。
4. 系统调用豆包完成需求结构化。
5. 系统调用豆包完成样片或图片/视频理解，并自动落标签。
6. 系统根据评分规则输出可解释评分。
7. 系统生成 Top 5 创意方向。
8. 创意团队选择方向并生成 4-5 个故事大纲或梗概。
9. 系统调用 `gpt-image-2-all` 生成氛围图并保存到 OSS。
10. 商务团队编辑提案、报价和合同。
11. 系统保存合同历史快照并导出 PDF/Word。
12. 商务团队创建飞书文档并发送给甲方个人或群。
13. 系统回写飞书文档链接、发送状态和归档记录。
14. 刷新页面或重新打开项目后，所有阶段状态和产物都能恢复。

### 14.2 权限验收

- 商务团队不能配置评分规则。
- 创意团队不能编辑报价和合同。
- 非管理团队不能查看全局模型成本和敏感审计数据。
- 管理团队可以查看全局项目、异常任务、成本和规则配置。
- API 层必须拒绝越权操作，不能只依赖前端隐藏按钮。
