# 全流程界面卡片检查项目设计规格

## 背景

当前已有项目 `流程巡检 / 全流程界面卡片检查 2026-6-25 20:50:55`，但它只具备部分阶段状态，缺少各 SOP 主要业务产物。结果是：即使阶段节点能点开，很多界面卡片仍然是空状态，无法稳定检查卡片布局、按钮顺序、状态文案和审核入口。

本规格的目标是把这个项目整理成一个专用的 UI 巡检项目。它用于验收界面，不用于证明真实业务链路已全部执行完成。

## 目标

- 10 个 SOP 节点都可以进入并查看。
- 每个 SOP 节点至少有一组可展示的持久化样例数据。
- 样例数据刷新页面后仍然存在。
- 样例内容统一标记为 `UI 巡检样例`，避免和真实客户项目混淆。
- 不调用真实 AI，不触发真实图片/视频生成，不发送飞书消息。
- 不改变真实项目的业务流转规则。

## 项目状态设计

UI 巡检项目主状态应设置为最后一个或较后阶段，以便阶段导航允许回溯查看所有前置节点。

推荐：

- `projects.current_stage = settlement_delivery_archive`
- `projects.status = in_progress`
- 前 9 个 SOP 阶段的 `project_stage_states.status` 设为 `completed` 或 `approved`
- `settlement_delivery_archive` 设为 `in_progress`

这样用户可以从阶段导航回看所有节点，同时不会误以为真实项目已经完整归档。

## SOP 样例数据范围

### SOP 1 Brief 收集与需求结构化

需要展示：

- 客户原始输入摘要。
- 标准化 Brief。
- 缺失信息清单。
- Brief 已确认或待确认状态。

样例内容必须说明这是巡检数据，例如：`本 Brief 为 UI 巡检样例，用于检查字段布局。`

### SOP 2 风险体检卡

需要展示：

- 五个风险维度状态灯。
- 事实清单。
- 原文依据。
- 置信度。
- 人工判断区。

风险等级建议混合红、黄、绿，便于检查不同状态样式。

### SOP 3 两轮创意视觉提案

需要展示：

- 4 个创意方向。
- 至少 1 个已选方向。
- Round 1 氛围图样例。
- Round 2 深化提案样例。
- 甲方审核任务和反馈状态。

氛围图可以使用已有 OSS 样例图或稳定占位图记录，但不能伪装为真实 AI 新生成。

### SOP 4 工作量估算、报价合同与交付清单

需要展示：

- 工作量估算。
- 报价明细与总价。
- 报价历史版本。
- 合同内容与历史版本。
- 交付清单。
- 飞书交付记录样例。
- 最终签署确认按钮状态。

飞书交付记录应标记为样例，不实际发送消息。

### SOP 5 脚本、人物场景设定与文字分镜确认

需要展示：

- 完整剧本。
- 剧本格式检查结果。
- 人物清单。
- 场景清单。
- 每个人物/场景的 Prompt 文本框。
- 设定图候选图。
- 文字分镜序列。

人物/场景设定图候选可以引用已有成功生成图片，或写入样例图片 URL。

### SOP 6 分镜图片生产与三批审核

需要展示：

- 场次列表。
- 分镜内容描述。
- 当前分镜主图候选区。
- 候选图切换。
- 右侧全部分镜缩略图导航。
- 三批提报记录。
- 逐镜甲方反馈。

至少准备一个场次、三条分镜、每条分镜一张候选图。

### SOP 7 AI 视频生成与导演下发

需要展示：

- 已确认分镜图片缩略图。
- 当前分镜内容。
- 视频候选播放区域。
- 多个视频版本卡片。
- Prompt、模型、数量控制。
- 按场次打包下载状态。

视频 URL 可以使用样例视频地址或空视频记录加状态文案，但必须让布局完整可检查。

### SOP 8 A-copy 生成与多轮修改

需要展示：

- A-copy 成片记录。
- 2 到 3 轮甲方反馈样例。
- 修改记录。
- 版本快照。
- 当前审核状态。

### SOP 9 B-copy 定稿确认与交付清单核对

需要展示：

- B-copy 最终版本。
- 最终确认状态。
- 交付清单核对项。
- 未完成、已完成、需补充等状态混合。

### SOP 10 结算交付与完整归档

需要展示：

- 费用回收状态。
- 最终交付包。
- 归档清单。
- 项目收尾记录。
- 归档操作区。

## 实现方式

推荐新增一个内部脚本或内部管理动作，职责是为指定项目补齐 UI 巡检样例数据。

建议脚本名：

```bash
npm run seed:ui-inspection
```

脚本行为：

1. 查找现有 `全流程界面卡片检查` 项目。
2. 如果不存在，则创建一个新的 `流程巡检 / 全流程界面卡片检查` 项目。
3. 写入或更新 10 个 SOP 阶段状态。
4. 写入每个阶段需要的最小样例业务产物。
5. 保持幂等：重复运行不会无限创建重复卡片。
6. 输出项目 ID 和下一步访问提示。

## 数据原则

- 样例数据必须持久化到真实数据库。
- 样例数据使用真实表结构，不引入前端内存 mock。
- 样例数据必须有稳定标识，例如标题或 metadata 中包含 `ui_inspection_sample`。
- 重复运行脚本时，应优先复用已有样例记录，必要时更新内容。
- 不删除真实用户创建的数据。
- 不覆盖非巡检项目。

## UI 验收方式

验收时用户应能：

1. 在项目列表中打开 `全流程界面卡片检查`。
2. 点击顶部 10 个 SOP 节点。
3. 每个节点都看到主要卡片，而不是大片空白。
4. 检查按钮文案、排序、折叠状态、审核入口、版本快照和错误/空/成功状态。
5. 刷新页面后仍能继续检查。

## 不做范围

- 不真实调用 AI 生成新内容。
- 不真实发送飞书消息。
- 不真实跑完整项目交付。
- 不修改真实项目的阶段流转规则。
- 不把巡检样例当作客户项目完成证明。
- 不为巡检引入新的前端页面。

## 成功标准

- 巡检项目主阶段和阶段明细一致，不再停留在早期节点。
- 10 个 SOP 节点都可进入。
- 每个节点至少一张核心卡片有内容可检查。
- 主要审核卡片、版本卡片、候选图区、分镜导航、视频候选区、归档区都能看到。
- 运行脚本后不需要手动改数据库。
- 项目刷新后数据仍然存在。
