# SOP 2 风险体检卡简洁面板设计

日期：2026-06-30

## 背景

当前 SOP 2 风险体检卡已经具备后端生成、人工决策和状态机流转能力，但前端呈现过于分散：

- 风险维度、关键依据、低置信度项被拆成大量小方框，用户需要自己拼出结论。
- “关键依据”逐条展开后信息量过重，不利于快速判断接不接。
- 决策按钮没有沿用品牌主按钮规范，`能接` 被做成绿色按钮，削弱了 MOPHRO 当前工作台的品牌蓝主操作体系。

这次设计只调整 SOP 2 风险体检工作区的呈现方式和前端派生展示逻辑，不改变已经确认的后端决策状态机。

## 目标

把 SOP 2 风险体检卡改成一整块“风险结论面板”，让用户一眼看到：

1. 当前总体能不能推进。
2. 哪些点会影响接单。
3. 下一步只能做 `可以接` 或 `不可以接`。

界面要比当前版本更简洁，重点是决策辅助，不是展示模型抽取的全部事实。

设计方向必须高度可视化、更清晰、更简洁、更精准。凡是不能帮助用户完成当前判断的信息、文字、容器和流程装饰，都默认删除。

## 非目标

- 不修改风险体检卡生成接口。
- 不修改 `accept` / `reject` / `brief_insufficient` / `project_blocked` 的后端状态机规则。
- 不删除后端保存的 facts、dimensions、redline alerts 数据。
- 不新增新的人工决策分支。
- 不做独立详情页。
- 在 SOP 2 风险体检工作区内，不展示无意义的笼统步骤条或自动编号，例如 `1/2/3/4 步`、`SOP1~7`、`5 个主步骤`、`几 CP`、`三批` 等与当前风险判断无关的流程装饰。

## 交互结构

### 空状态

未生成风险体检卡时，保留当前空状态和 `生成风险体检卡` 按钮。

文案保持自然语言：

- “还没有风险体检卡。请先完成 Brief 结构化，再生成订单落地可行性评估和人工决策区。”

### 已生成状态

风险卡生成后，工作区主体只展示一个主面板：

1. 顶部结论区
   - 标题：`风险体检卡`
   - 总结论：根据 `overallAlert` 显示 `可推进`、`需要补充确认`、`落地风险偏高`、`命中红线`
   - 当下状态：只显示当前风险判断状态，例如 `待人工判断`、`已通过`、`已退回补资料`、`已阻塞`
   - 一句话摘要：用 1 句说明主要风险集中在哪里
   - 不显示 SOP 编号、步骤条、阶段数量或跨阶段流程装饰

2. 风险卡点区
   - 只展示“不可行点 / 风险卡点”
   - 最多 5 条
   - 每条包含：
     - 标题，例如 `授权边界不清`
     - 风险等级，例如 `高风险`、`中风险`、`待确认`
     - 简短原因，一句话即可
   - 不再展示“关键依据”逐条事实清单
   - 不再展示所有低风险或正常维度

3. 决策区
   - 位于主面板底部
   - 两个按钮并列，沿用现有 Button 规范
   - `可以接`：品牌蓝主按钮，使用默认 `Button` variant
   - `不可以接`：destructive / outline 风格，避免绿色大块和风格跳脱

4. 不可以接抽屉
   - 保持现有线性流程：
     - 点击 `不可以接`
     - 选择 `Brief 不足` 或 `项目背景/项目本身原因`
     - 填写 `理由补充`
     - 点击确认
   - 抽屉只在点击 `不可以接` 后出现，不常驻在主面板。

## 风险卡点派生规则

前端从现有风险卡数据派生最多 5 条风险卡点，不新增数据库字段。

优先级：

1. 红线告警
   - 每条 `redlineAlerts` 都可成为风险卡点。
   - 等级显示为 `红线` 或 `高风险`。

2. 高风险维度
   - `dimensions.level === "high"` 的维度进入候选。
   - 标题使用维度中文名，例如 `素材可用性不足`、`合规与交付风险偏高`。
   - 原因优先取 `dimension.evidence`，没有 evidence 时取 `anchorText`。

3. 中风险维度
   - 当红线和高风险不足 5 条时补充 `medium` 维度。
   - 标题使用维度中文名。

4. 低置信度关键缺口
   - 当仍不足 5 条时，从 facts 中选择 `confidence < 0.65` 且 value 显示为 `未说明`、`待人工确认` 或 evidence 为空的项。
   - 显示为 `待确认`。

去重：

- 同一维度只出现一次。
- 同名 fact 和 dimension 同时出现时，优先展示 dimension。

无风险卡点：

- 显示单条空提示：“当前没有明确阻塞点，仍建议人工确认后推进。”

## 文案规则

按钮文案：

- 主按钮：`可以接`
- 次按钮：`不可以接`

顶部摘要示例：

- 红线：`当前命中红线风险，请先确认是否具备继续承接条件。`
- 高风险：`当前主要卡点集中在授权、预算、交付规格等落地条件。`
- 中风险：`当前有少量信息需要确认，不影响人工判断但建议接单前补齐。`
- 低风险：`当前没有明显阻塞点，可以进入人工接单判断。`

风险卡点原因必须简短。前端可做截断或 line-clamp，避免长段 evidence 撑开面板。

## 视觉规范

采用现有工作台设计系统，不新增独立视觉语言。

- 主面板使用现有 `WorkspaceCard` / `ds-card` 风格。
- 圆角、边框、阴影沿用 CSS 变量。
- 品牌色使用 `--accent` / `--primary`，即当前 MOPHRO 蓝 `#207fec`。
- `可以接` 使用默认主按钮，不使用绿色背景。
- `不可以接` 使用 destructive outline 或轻底色风格。
- 风险卡点可以用左侧细色条、等级 pill、图标或色块做高度可视化表达，但不做大量嵌套卡片。
- 不需要展示的辅助标题、解释文字、装饰容器一律删除。
- 页面视觉重心必须落在“总判断”和“不可行点”上，按钮区紧跟在同一块面板底部。

## 状态与权限

- `可以接` 仅商务团队和管理员可保存。
- `不可以接` 仅商务团队和管理员可保存。
- 创意团队仍可生成或查看风险卡，但不能保存接单结论。
- loading、success、error 反馈沿用当前自然语言提示。

## 数据流

1. 用户点击 `生成风险体检卡`。
2. 前端调用现有 `generateRiskCheck(project.id)`。
3. 后端返回 `riskCheck`，页面刷新工作区数据。
4. 前端从 `riskCheck.redlineAlerts`、`riskCheck.dimensions`、`riskCheck.facts` 派生 `riskIssues`。
5. 用户点击 `可以接`：
   - 调用 `saveRiskCheckDecision({ decision: "accept" })`
   - 后端推进到 SOP 3
6. 用户点击 `不可以接`：
   - 打开抽屉
   - 提交 `rejectionCategory` 和 `reason`
   - 后端按现有规则回退 SOP 1 或阻塞 SOP 2

## 测试与验收

需要更新 baseline 测试，保护以下行为：

- SOP 2 主面板不再出现 `关键依据` 大列表。
- 主面板出现 `可以接` 和 `不可以接`。
- 不出现绿色 `能接（通过）` 文案。
- 不出现 `1/2/3/4 步`、`SOP1~7`、`5 个主步骤` 这类笼统步骤装饰。
- 风险卡点最多 5 条。
- `不可以接` 后仍出现 `Brief 不足`、`项目背景/项目本身原因`、`理由补充`。
- 后端风险决策 baseline 保持通过。

验证命令：

```bash
npm run test:baseline
npm run typecheck
npm run lint
npm run build
```

涉及 UI 时，需要用浏览器检查：

- 未生成风险卡空状态。
- 已生成风险卡主面板。
- `不可以接` 抽屉。
- 桌面与窄屏按钮不溢出。

## 验收标准

- 用户不需要阅读事实清单，就能看到当前最重要的不可行点。
- 页面上不再有大量小方框堆叠。
- 页面不再展示与当前判断无关的步骤条、编号、阶段装饰和冗余容器。
- `可以接` / `不可以接` 处在同一个主面板底部，视觉上是主流程的一部分。
- 品牌蓝回到主操作按钮。
- 后端决策逻辑和状态机不回退。
