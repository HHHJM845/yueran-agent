# 子功能任务卡片统一设计规范

## 目标

将工作台正文区域里的子功能卡片统一成一套“极简任务卡系统”。这套规范服务六大功能模块下的业务卡片和常见内部小块，让页面保持当前冷色清爽视觉系统，同时让用户一眼看清：这是什么、当前是否正常、下一步做什么、结果在哪里看。

## 非目标

- 不改变三栏工作台布局。
- 不改变顶部六大功能导航和阶段状态机。
- 不改变业务流程、API、数据库结构或持久化规则。
- 不重新设计整套工作台控件。
- 不引入新的主色、品牌色或大面积装饰风格。

## 视觉基线

继续沿用项目当前全局设计变量：

- 页面背景：`--workspace-background: #E5E7E9`
- 卡片与侧边栏：`--surface-card: #F9FBFC`
- 主强调色：`--accent: #207FEC`
- 文字主色：`--text-primary`
- 说明文字：`--text-secondary`
- 弱说明文字：`--text-tertiary`
- 边框：`--border-soft`
- 圆角：`--radius-card`、`--radius-card-sm`、`--radius-control`
- 阴影：`--shadow-card`、`--shadow-pill`

强调色只用于当前选中态、可继续推进的轻亮状态、按钮焦点、阶段点和关键进度。红色只用于失败、阻塞、警示或不可恢复风险。

## 卡片结构

每张子功能卡片统一为四层：

1. 卡片标题行
2. 一句短说明
3. 主体内容
4. 操作区

标题行包含：

- 左侧：线性图标加标题。
- 右侧：轻状态胶囊、主按钮或必要操作。

短说明规则：

- 控制在一到两行。
- 只说明当前卡片的用途和下一步价值。
- 不重复项目基础信息、阶段说明或按钮文案。
- 能由状态或按钮表达的信息，不再写成长段说明。

主体内容按业务类型放置表单、上传、编辑器、结果列表或空状态。操作区只有在需要多个按钮时出现；单一主操作优先放在标题行右侧。

## 字体层级

卡片标题使用中等加粗，优先贴近 `--text-section-title-size` 或更紧凑的卡片标题层级。

说明文字使用 `--text-body-size` 和 `--text-secondary`，行高保持轻松但不铺张。

字段标签使用 `--text-label-size`，颜色不超过 `--text-primary`，避免和标题抢层级。

状态数字、版本号和计数信息使用小胶囊，例如 `v0`、`草稿`、`0 条分镜`，不能成为页面视觉主角。

## 状态规则

正常状态保持轻量：

- 使用右上角小胶囊或小圆点。
- 可以显示 `草稿`、`已保存`、`0 条分镜`、`v1` 等短状态。
- 普通进行中不大面积强调。

异常状态才明显提示：

- `failed`
- `blocked`
- `needs_revision`
- `waiting_review`
- 不可操作或缺少前置条件

异常提示必须说明发生了什么和下一步可做什么，不直接展示 HTTP、数据库、SDK 或模型原始错误。

## 内部小块

常见内部小块统一为“轻边框、浅灰底、充足内边距、无厚阴影”。

适用对象：

- 输入框组
- 上传区
- 飞书或外部链接录入区
- 参考图小卡
- 结果列表
- 空状态
- 版本快照摘要
- 文字分镜行项目

内部小卡不要再制造过多层级。默认使用浅灰底和细边框；只有可点击、当前选中或异常状态才使用强调色或明显边框。

## 空状态

空状态统一为一句自然语言：

- 当前没有什么。
- 用户下一步应该做什么。

示例：

- “暂无文字分镜。保存完整剧本后点击自动拆分，系统会生成分镜并写入数据库。”
- “暂无参考图。先补充标题、风格标签和描述，再保存脚本方向包。”

空状态不使用大面积插画，不使用过强装饰色。

## 按钮规则

每张任务卡最多一个主按钮。

主按钮用于：

- 生成
- 保存
- 提交
- 创建审核链接
- 发送
- 刷新关键结果

次要按钮使用 outline 或 ghost，不比主按钮更抢眼。危险动作使用 destructive，不复用蓝色强调色。

按钮位置优先级：

1. 标题行右侧
2. 卡片底部右侧
3. 移动端自然换行到标题下方或底部

按钮必须有 loading、disabled、success 和 error 反馈，避免重复提交。

## 信息压缩规则

主视图只保留：

- 标题
- 一句短说明
- 轻状态
- 当前主体内容
- 主操作

以下内容应进入二级区域、折叠区或次级列表：

- 历史快照
- 诊断说明
- 长提示
- 版本详情
- 已归档记录
- 非当前步骤的辅助说明

项目基础信息与阶段说明不得在上方重复铺一层。如果信息已经能在卡片内部表达，上方说明应移除或合并。

## 适用范围

本规范适用于工作台正文区域的子功能卡片和常见内部小块：

- 项目基础信息
- 项目资料中心
- 需求结构化
- 资料解析与标签评分结果
- 技术不可行或阻塞管理
- Top 5 创意方向
- Agent 商务文档草稿
- 提案编辑与版本快照
- 报价编辑与版本快照
- 合同模板填充与版本快照
- 飞书交付闭环
- 脚本创意与文字分镜确认
- 参考图小卡
- 文字分镜结果
- 分镜图片自由画布中的常见子卡
- AI 视频自由画布中的常见子卡
- A copy、B copy 和归档阶段的后续预留卡片

## 实现边界

后续实现应优先新增或收敛可复用 UI 结构，而不是在每个模块里继续临时拼样式。

建议抽象：

- `TaskCard`
- `TaskCardHeader`
- `TaskCardStatus`
- `TaskCardActions`
- `TaskFieldGroup`
- `TaskEmptyState`
- `TaskResultList`

这些结构只负责视觉和布局，不承载业务状态机逻辑。业务状态仍从现有后端、项目阶段状态和真实产物数据读取。

## 验收标准

- 同一屏里的子功能卡片标题、说明、状态、按钮位置保持一致。
- 正常状态轻量，异常和待确认状态足够明显。
- 主按钮数量受控，每张卡最多一个主按钮。
- 长说明被压缩，历史、诊断和版本详情不挤占主视图。
- 输入框组、上传区、参考图小卡、空状态、结果列表使用统一轻量容器风格。
- 卡片在桌面和移动视口下不发生文字溢出、按钮重叠或横向撑开。
- 不改变三栏布局、顶部六大功能导航、状态机和业务接口。
- UI 改动后通过 typecheck、lint、build，并完成浏览器检查。
