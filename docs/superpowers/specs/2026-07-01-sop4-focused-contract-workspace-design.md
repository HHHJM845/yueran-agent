# SOP4 聚焦式合同报价与交付清单工作区设计

日期：2026-07-01

## 背景

当前 SOP4「工作量估算、报价合同与交付清单」工作区会一次性展示合同模板上传、工作量估算、商务草稿、报价、合同、交付清单等多个区域。页面视觉优先级混乱，用户不知道当前应该处理哪件事，也违背了“工作区只聚焦当前阶段”的原则。

本设计只调整 SOP4 工作区展示与当前任务判断，不改变已有阶段键、数据库状态机、上传链路、报价保存、合同保存、交付清单保存等真实业务能力。

## 目标

1. 上传合同模板前，主体工作区只显示合同模板投放区。
2. 上传合同模板后，主体工作区只显示当前需要处理的一个业务任务。
3. 当前任务由持久化数据自动判断，刷新页面后仍恢复到同一任务。
4. 不再把工作量、报价、合同、交付清单、锁定等内容同时堆在页面里。
5. 保留很轻的进展提示，但不默认展开历史内容。

## 非目标

1. 不重写后端报价、合同、交付清单的保存模型。
2. 不新增新的 SOP 阶段键。
3. 不做真正的 DOCX 内容解析 AI；当前仍使用已上传合同模板资产作为流程启动条件和解析展示来源。
4. 不改变 SOP3、SOP5 或其他工作区。

## 工作区状态

### 上传前

当项目没有可识别的合同模板资产，且合同没有绑定 `clientContractAssetId` 时：

- 只展示 `ContractTemplateIntakeCard`。
- 不展示工作量估算、商务草稿、报价、合同、交付清单、锁定。
- 上传成功后刷新工作区，并自动进入上传后的当前任务判断。

合同模板资产识别沿用现有规则：

- `contract_file` 类型；
- 或文档资产文件名 / URL 包含合同、模板、协议、签约、报价、contract、template、agreement、quote 等关键词。

### 上传后

上传后主体区域根据数据完成度自动显示一个当前任务：

1. **工作量估算生成商务草稿**
   - 条件：存在合同模板资产，但没有 `workloadEstimate`。
   - 展示：`WorkloadEstimateCard`。

2. **报价编辑与甲方确认**
   - 条件：存在 `workloadEstimate`，但没有 `quote`。
   - 展示：`QuoteEditorCard`。
   - 商务草稿生成只作为报价前的辅助入口，不再作为常驻独立大区域。

3. **合同编辑和签署**
   - 条件：存在 `quote`，但没有 `contract`，或合同状态不是 `confirmed/sent/signed`。
   - 展示：`ContractEditorCard`。
   - 合同模板解析结果保留在合同编辑区内，但只服务当前合同任务。

4. **确认交付清单**
   - 条件：合同已经进入 `confirmed/sent/signed`，但没有 `deliveryChecklist`。
   - 展示：`DeliveryChecklistCard`。

5. **锁定**
   - 条件：存在已确认或已签署合同，并且存在交付清单。
   - 展示：一个简洁的锁定结论区域，说明 SOP4 已具备进入 SOP5 的条件。

## 视觉结构

SOP4 主体分为三层：

1. 当前任务标题区
   - 显示当前任务名称、状态和一句自然语言说明。
   - 不显示无意义编号、笼统步骤条或多个并列卡片标题。

2. 当前任务操作区
   - 只渲染当前任务对应的一个主组件。
   - 主按钮、错误、成功、空状态仍保留在当前组件内。

3. 轻量进展条
   - 页面底部显示五个短标签：模板、估算、报价、合同、清单/锁定。
   - 只用于提示已完成、当前、未开始。
   - 默认不展开历史内容。
   - 如后续要支持回滚或历史预览，应以小弹层或只读抽屉呈现，不回到主体堆叠。

## 状态判断规则

新增一个纯前端视图模型函数，例如 `createSop4FocusedFlowViewModel(input)`，输入来自真实工作区数据：

- `assets`
- `contract`
- `workloadEstimate`
- `quote`
- `deliveryChecklist`

输出：

- `templateAsset`
- `currentTask`
- `progressNodes`
- `showTemplateOnly`
- `summary`

`currentTask` 枚举：

- `upload_template`
- `workload_estimate`
- `quote_confirmation`
- `contract_signing`
- `delivery_checklist`
- `locked`

该函数只负责展示判断，不写数据库，不改变状态机。

## 数据与后端一致性

- 合同模板上传继续使用现有真实上传链路：创建上传 URL、OSS PUT、注册资产。
- 当前任务判断只使用后端返回的持久化数据，不依赖前端内存。
- 保存工作量、报价、合同、交付清单后，仍通过 `onRefresh` 获取最新工作区数据，再重新计算当前任务。
- 合同状态推进仍由现有 `saveProjectContract` 和状态机处理。

## 错误与空状态

- 上传失败：保留自然语言错误，说明可能原因和下一步。
- 上传成功但资产未识别为合同模板：显示“已上传，但未识别为合同模板，请确认文件名或重新选择合同/报价/协议类文档”。
- 当前任务缺少前置数据：不显示后续区域，展示一句可执行提示。

## 测试范围

1. SOP4 上传前只显示合同模板投放区。
2. 上传后无工作量估算时只显示工作量估算。
3. 有估算无报价时只显示报价。
4. 有报价无合同确认时只显示合同编辑。
5. 合同确认后无清单时只显示交付清单。
6. 合同和清单都存在时显示锁定。
7. `npm run test:baseline` 继续通过。
8. UI 改动完成后做浏览器检查，确认 SOP4 页面没有控制台错误、文字溢出或多区域堆叠。

## 验收标准

1. 用户进入 SOP4 后，一眼只看到当前该做的事情。
2. 上传合同模板前，后续区域完全不出现。
3. 上传合同模板后，系统自动切到当前任务。
4. 工作区不再同时展示多个大卡片。
5. 页面刷新后当前任务不丢失。
6. 相关测试、类型检查、lint、build 通过。
